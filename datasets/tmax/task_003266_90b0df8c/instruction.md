You are a support engineer collecting diagnostics for a distributed database that recently suffered a catastrophic crash. The only intact data we have are fragmented Write-Ahead Logs (WAL) generated by various nodes across different timezones. 

To recover the final state of the database, we normally use a legacy utility located at `/app/db_recover`. This compiled binary reads diagnostic WAL text lines from standard input (`stdin`) and outputs the final, resolved state of the database to standard output (`stdout`). 

Unfortunately, `/app/db_recover` is a stripped, black-box executable and we have lost its source code. We need to port this tool to Python to add it to our modern recovery pipeline. 

Your task is to write a Python script at `/home/user/recover.py` that behaves **exactly** like the `/app/db_recover` binary for any given valid input.

**Input Format (from stdin):**
The input consists of multiple lines. Each line represents a logged transaction event:
`[TxID] [LocalTimestamp] [TZ_Offset] [Operation] [Key] [Value]`

- `TxID`: Integer transaction ID.
- `LocalTimestamp`: Integer representing the local time in seconds.
- `TZ_Offset`: A timezone offset string (e.g., `+0530`, `-0400`).
- `Operation`: Either `SET` or `DEL`.
- `Key`: Alphanumeric string.
- `Value`: Alphanumeric string (ignored if Operation is `DEL`).

**Concurrency and Timezone Rules:**
You will need to deduce the exact timestamp calculation, concurrency resolution, and output formatting rules by experimenting with the `/app/db_recover` binary. Pass different timezone offsets and conflicting transactions to the binary to reverse-engineer how it decides which operation "wins" when multiple transactions target the same key.

**Requirements:**
1. Create the Python script at `/home/user/recover.py`.
2. It must read from `stdin` and print to `stdout`.
3. Its output must be bit-for-bit identical to `/app/db_recover` for any valid sequence of logs. 
4. Ensure the script efficiently handles large inputs and edge-case timezone boundaries.

You may use standard Linux tools (like `strings`, `gdb`, `objdump`, or shell redirection) to analyze the binary and interactively debug its logic.
