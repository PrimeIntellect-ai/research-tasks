You are a security engineer responsible for rotating the master credentials for a legacy physical access control system. The legacy system's key synchronization device only outputs cryptographic synchronization logs visually. We have captured the latest synchronization session in a video file located at `/app/credential_sync.mp4`.

Your task is to extract the cryptographic parameters from this video, derive the current master key using known-plaintext analysis, generate the new rotated key, and host it on an internal endpoint so the downstream authenticators can fetch it.

**Step 1: Extract and Decode**
1. The video `/app/credential_sync.mp4` displays a sequence of QR codes, updating exactly once per second. Extract these QR codes (you may use `ffmpeg` to extract frames and `zbarimg` to read the QR codes).
2. Each QR code contains a Base64 encoded payload.
3. Once Base64 decoded, each payload is a text string in the following format:
   `PT:<16-character-hex-plaintext>|CT:<16-character-hex-ciphertext>|HASH:<64-character-hex-sha256sum>`

**Step 2: Integrity Verification & Cryptanalysis**
1. **Integrity Verification:** For each decoded string, verify the file/payload integrity. Compute the SHA256 hash of the `<16-character-hex-ciphertext>` string (without any trailing newlines). If it does not exactly match the provided `HASH` value, discard that payload as corrupted.
2. **Cryptanalysis:** The legacy system uses a simple XOR cipher. For the valid payloads, the ciphertext was generated by XORing the plaintext with an 8-byte (16 hex character) Master Key. Perform a known-plaintext attack to derive the Master Key. (All valid payloads will yield the exact same 16-character hex Master Key).

**Step 3: Key Rotation & Service Hosting**
1. The rotated key is defined as the SHA256 hash of the 16-character hex Master Key. Calculate this by passing the derived 16-character hex string (with a trailing newline) into `sha256sum`.
2. Create a purely Bash-based network service (using `nc` or `socat` in a loop) that listens on `127.0.0.1:9090`.
3. The service must act as an HTTP server. When it receives a `GET /rotate HTTP/1.1` request, it must respond with:
   ```
   HTTP/1.1 200 OK
   Content-Type: application/json

   {"new_key": "<calculated_rotated_key>"}
   ```
   *(Note: Ensure correct HTTP line endings, `\r\n`, and an empty line before the JSON body).*

Leave this Bash server running in the background. The automated verifier will make an HTTP GET request to `http://127.0.0.1:9090/rotate` to validate your completion of the task.
