You are a platform engineer tasked with unblocking a legacy CI/CD pipeline that has failed due to bitrot, broken build scripts, and a lost configuration file. Your objective is to recover the build configuration from a video artifact, fix the broken build system, and implement a replacement for a missing stream processor.

**Step 1: Recover the Build Configuration**
The previous pipeline stored its environment configuration in a video artifact generated by an old diagnostic tool. You are provided with this video at `/app/build_signal.mp4`. 
- The video resolution is 200x200.
- It encodes a binary string using the color of the center pixel (x=100, y=100).
- Pure black `#000000` represents `0`. Pure white `#FFFFFF` represents `1`.
- The video runs at 10 frames per second. Each frame corresponds to exactly 1 bit.
- Parse the bits sequentially from the first frame to the last. Every 8 bits form an ASCII character (Most Significant Bit first). 
- Extract this ASCII string. You will need it to verify your state machine later.

**Step 2: Fix the Build System**
In `/home/user/pipeline_tools/`, there is a C project with a broken `Makefile`. 
- The project builds two shared libraries, `libalpha.so` and `libbeta.so`, and a main executable `pipeline_test`.
- Currently, the build fails because the Makefile has a circular dependency setup and incorrectly manages the Application Binary Interface (ABI) linking flags.
- Your task is to repair the `Makefile` and any minimal C headers necessary so that running `make` successfully produces both shared libraries and the `pipeline_test` executable without warnings or errors. Do not change the core logic of the `.c` files, only fix the compilation, linking process, and header definitions.

**Step 3: Implement the Stream Processor**
A core component of the pipeline is a state-machine parser that sanitizes log output. The original executable is lost. You must write a replacement executable at `/home/user/stream_processor` (in any language, but it must be an executable file, e.g., a script with a shebang and `chmod +x`, or a compiled binary).
- It must read arbitrary text from `stdin` and write to `stdout`.
- It must implement a state machine that performs HTML escaping for three specific characters: `<` to `&lt;`, `>` to `&gt;`, and `&` to `&amp;`.
- **Crucial Rule:** The escaping must ONLY happen *outside* of inline code blocks. 
- An inline code block is toggled by a backtick character (`` ` ``). The backtick itself is always printed. Any `<`, `>`, or `&` characters appearing while the code block state is active must be printed exactly as-is, without escaping. A second backtick turns the code block state off.
- The stream processor must handle inputs of any size efficiently.
- Ensure the executable operates purely on standard I/O streams and exits cleanly when `stdin` is closed.

Complete all steps. Your final `stream_processor` will be rigorously tested against an automated fuzzer comparing it to a reference oracle.
