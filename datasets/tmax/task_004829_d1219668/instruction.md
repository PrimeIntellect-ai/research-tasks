You are assisting a researcher who is organizing a large dataset of sensor logs. The logs are being streamed from various field devices and are periodically uploaded as zip archives. 

Your task is to create a network service that processes these uploads securely, parses proprietary sensor logs using an undocumented legacy binary, and renames the files according to their internal metadata.

Here are the exact requirements:

1. **Service Endpoint:**
   Create an HTTP server listening exactly on `127.0.0.1:9000`. It must accept `POST` requests to the `/upload` endpoint, where the request body contains the raw bytes of a zip archive.

2. **Secure Extraction (Zip Slip Protection):**
   The uploaded zip archives contain `.wal` (Write-Ahead Log) files. However, some archives generated by malfunctioning field devices (or a rogue intern) contain malformed paths designed to overwrite system files (e.g., `../../../../home/user/malicious.txt`). 
   Your service must extract the contents of the uploaded zip to a temporary directory (e.g., under `/tmp/dataset_staging/`), but it MUST strictly prevent path traversal. Any file in the archive that attempts to extract outside the intended temporary directory must be completely ignored or skipped.

3. **Log Parsing using the Legacy Decoder:**
   We have an old, stripped proprietary binary located at `/app/sensor_decoder`. You must use this binary to analyze every successfully extracted `.wal` file. 
   When executed with a `.wal` file path as its first argument (e.g., `/app/sensor_decoder /path/to/file.wal`), it outputs a multi-line text record. You will need to reverse-engineer or black-box test this binary to understand its output format and figure out how to parse the `SensorID` from its output.

4. **Bulk Renaming:**
   For each extracted `.wal` file, extract the `SensorID` using the decoder. Rename the file in the staging directory to follow the format: `<SensorID>_<OriginalFileName>.wal`.
   For example, if the original file was `data_01.wal` and the decoder reveals the SensorID is `X7-992`, rename it to `X7-992_data_01.wal`.

5. **Response Format:**
   After processing the archive, the HTTP POST response must return a `200 OK` status code. The response body must be a valid JSON array of strings containing the final, renamed basenames of all successfully processed `.wal` files (e.g., `["X7-992_data_01.wal", "B2-111_log.wal"]`). Ignore any files that were not `.wal` files or were skipped due to path traversal attempts.

You may use Python, Bash, or any standard Linux utilities available in your environment to build this service. Ensure your service runs continuously and handles multiple sequential requests.
