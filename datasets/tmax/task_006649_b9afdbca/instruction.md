You are a systems programmer tasked with building a bash-based automated CI/CD debugging pipeline to detect C library linking incompatibilities and memory leaks. The system uses a set of log files generated by a previous build step.

You need to implement the following components entirely in Bash.

**Phase 1: Semantic Version Checker (`/home/user/check_versions.sh`)**
A mock `ldd` output file is located at `/home/user/lib_versions.txt`. Each line contains a library name and its resolved version, formatted as `lib_name.so - MAJOR.MINOR.PATCH` (e.g., `libssl.so - 1.1.1`).
Write a script that reads this file and finds the version of `libmath_custom.so`. 
Mathematically compare its version against the required minimum version of `2.4.1`. A version A is less than B if A's MAJOR is less, or MAJOR is equal and MINOR is less, etc.
- If the resolved version is less than `2.4.1`, output exactly: `LINK_ERROR`
- If it is `2.4.1` or greater, output exactly: `LINK_OK`

**Phase 2: Memory Profile Math (`/home/user/parse_mem.sh`)**
A memory profiling log is located at `/home/user/valgrind_mock.log`. 
Lines indicating leaks follow this exact format: `[WARN] LEAK: <number> bytes at <hex_address>` (e.g., `[WARN] LEAK: 1024 bytes at 0x8048000`).
Write a script that extracts all the byte values from these specific lines, sums them up mathematically, and evaluates the total.
- If the total leaked bytes are strictly greater than 4096, output exactly: `MEM_FAIL: <total>`
- Otherwise, output exactly: `MEM_OK: <total>`

**Phase 3: Bash API Server (`/home/user/api.sh`)**
Write a script that acts as a simple REST API server using `nc` (netcat). It should listen on port `8080`.
When it receives an HTTP `GET /status` request, it should execute `/home/user/check_versions.sh` and `/home/user/parse_mem.sh`.
It must respond with a valid HTTP 200 OK response containing the following JSON body:
`{"version_status": "<output_from_check_versions>", "mem_status": "<output_from_parse_mem>"}`
*Note: Make sure to include proper HTTP headers (e.g., `HTTP/1.1 200 OK` followed by `Content-Length` and double CRLF).* It should handle exactly one request and then exit gracefully.

**Phase 4: CI/CD Integration (`/home/user/ci_pipeline.sh`)**
Write the main CI pipeline script that:
1. Starts `/home/user/api.sh` in the background.
2. Waits for 2 seconds to allow the server to start.
3. Uses `curl -s http://localhost:8080/status` to fetch the JSON response and saves it to `/home/user/ci_result.json`.

Ensure all your scripts are executable (`chmod +x`). Run `/home/user/ci_pipeline.sh` to generate the final output file before completing the task.
