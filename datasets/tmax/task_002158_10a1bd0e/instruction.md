We have a legacy web security architecture that relies on a proprietary token signing binary whose source code (originally written in Rust) has been lost. The binary is located at `/app/legacy_signer`. It is a statically-linked, stripped executable. 

When you run `/app/legacy_signer <payload>`, it outputs a custom-encoded security signature for that payload. Currently, our system shells out to this binary for every web request, which is causing severe performance bottlenecks.

Your objective is to replace this binary by writing a high-performance Python reverse proxy that natively reimplements the encoding logic.

Here are your requirements:
1. **Analyze the Binary**: Investigate `/app/legacy_signer` to understand its character and data encoding logic. (Hint: It performs a combination of string manipulation, a constant XOR cipher, and standard encoding). You can use the binary as an oracle to test your hypotheses.
2. **Implement Native Python Logic**: Translate the reverse-engineered logic into pure Python. You must not use `subprocess` or `os.system` to call the binary in your final solution, as this defeats the performance goal.
3. **Build a Reverse Proxy**: Write a Python script at `/home/user/proxy.py` that starts an HTTP reverse proxy on `localhost:8080`. 
    - All incoming `GET` requests must be forwarded to a backend server running at `http://localhost:9000`.
    - Before forwarding, your proxy must intercept the request and add a new HTTP header: `X-Custom-Signature`.
    - The value of `X-Custom-Signature` must be the encoded signature of the full request URI (path + query string, e.g., `/api/data?user=1`).

The automated verification system will start a dummy backend on port 9000, start your proxy, and blast it with a large number of requests. It will strictly evaluate your solution based on two metrics:
- **Accuracy**: 100% of the signatures generated by your proxy must exactly match what `/app/legacy_signer` would output for the same URI.
- **Performance Threshold**: Because you are natively implementing the logic in Python instead of shelling out, your proxy must process 2,000 requests in under 1.5 seconds. 

Ensure your proxy script is executable, runs indefinitely until interrupted, and correctly handles concurrent or rapid sequential requests.
