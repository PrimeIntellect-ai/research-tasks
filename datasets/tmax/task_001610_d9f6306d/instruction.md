You are tasked with building a Rust-based configuration tracking service that interfaces with a legacy, stripped proprietary binary. 

We have a legacy configuration writer binary located at `/app/config_writer`. This binary continuously writes configuration updates into a Write-Ahead Log (WAL) format in the `/home/user/wal_archive` directory. The binary writes these files as compressed streams (using standard GZIP compression but with a custom internal binary structure) and frequently rotates them. 

Your goal is to build a Rust application that acts as a configuration manager and parser. It must do the following:
1. Parse the compressed WAL files generated by `/app/config_writer`. 
2. Understand the custom binary structure inside the GZIP stream. You will need to reverse-engineer or observe the output of `/app/config_writer` to deduce the internal format. It writes simple key-value configuration pairs (where keys are null-terminated strings and values are 32-bit little-endian integers), prefixed by a timestamp and an operation code (insert/update/delete).
3. Track the latest state of the configuration by applying these WAL operations in order.
4. Expose an HTTP API on `0.0.0.0:8080` that allows querying the current configuration state.

The HTTP API must support the following endpoints:
- `GET /config/:key`: Returns an HTTP 200 with a JSON body `{"key": "string", "value": int}` representing the current value of the key. If the key does not exist or was deleted, return HTTP 404.
- `GET /status`: Returns an HTTP 200 with `{"files_processed": int}`.

To test your service, you should run the `/app/config_writer` in the background (it will daemonize and start producing WAL files) and start your Rust server.

Please write your Rust application in `/home/user/config_tracker`. You may use standard Rust crates (like `flate2` for decompression, `tokio`, `hyper`, `axum`, etc., for the HTTP server).
