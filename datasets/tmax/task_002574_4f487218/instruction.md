You are a data analyst tasked with building a robust ETL pipeline for processing messy time series data from IoT temperature sensors. 

We have a directory of raw CSV files at `/home/user/raw_data/`. These files are generated by different systems and have several inconsistencies:
1. **Varying Encodings:** The files are saved in different character encodings (`utf-8`, `utf-16`, and `iso-8859-1`).
2. **Inconsistent Schemas:** 
    - File 1 has columns: `Date`, `Temp`, `Metadata`
    - File 2 has columns: `Timestamp`, `Temperature`, `Info`
    - File 3 has columns: `Time`, `Temp`, `Meta`
3. **Embedded Structured Data:** The device identifier is trapped inside a free-text metadata column. 
    - Format A (in Metadata/Meta): `"device_id=S1; battery=90"`
    - Format B (in Info): `"device_id: S2 | battery: 85"`
4. **Different Date Formats:** 
    - `YYYY-MM-DD HH:MM:SS`
    - `MM/DD/YYYY HH:MM AM/PM`
    - `YYYY-MM-DDTHH:MM:SSZ`
5. **The Retry Problem:** File 3 contains some data that overlaps perfectly with File 1 (simulating a failed job that was retried). If we blindly append data, we will get duplicate records.

**Your Goal:**
Write and execute a Python script at `/home/user/process_data.py` that processes all CSV files in `/home/user/raw_data/` and loads them into a SQLite database at `/home/user/db/time_series.db`.

**Requirements:**
1. Read all CSV files, handling their respective encodings correctly (you can use `chardet` or try common encodings).
2. Extract the `device_id` (e.g., `S1`, `S2`) from the metadata/info column.
3. Normalize all timestamps to standard ISO 8601 format without a timezone indicator: `YYYY-MM-DD HH:MM:SS`.
4. Create a SQLite table named `readings` with the following schema:
   - `timestamp` (TEXT)
   - `device_id` (TEXT)
   - `temperature` (REAL)
5. **Deduplication:** Ensure that the combination of `timestamp` and `device_id` is strictly unique in the database. When the script encounters a duplicate reading for the same timestamp and device, it should ignore the new duplicate and keep the existing record. Make sure your database schema enforces this constraint.
6. The script should be idempotent (running it multiple times should result in the exact same database state without errors).

Run your script to populate the database.
